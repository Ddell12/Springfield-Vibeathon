# Clerk → Convex Auth Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace all Clerk authentication with `@convex-dev/auth`, giving email+password, Google, and Apple sign-in, with roles stored directly in the Convex `users` table.

**Architecture:** The Clerk JWT bridge (`ConvexProviderWithClerk`, `clerkMiddleware`, `auth()`) is removed entirely. `ConvexAuthProvider` handles both auth and Convex wiring. Roles move from Clerk `publicMetadata` (JWT claim) into a `role` field on the `users` table, read via `ctx.db` instead of JWT parsing. The sign-in card is rewritten from an email-code OTP flow to an email+password flow with OAuth buttons.

**Tech Stack:** `@convex-dev/auth` v0.0.x, `@auth/core@0.37.0`, Password provider, Google provider, Apple provider, `convexAuthNextjsMiddleware`, Vitest + convex-test, Playwright E2E.

**Spec:** `docs/superpowers/specs/2026-04-03-clerk-to-convex-auth-design.md`

> **⚠️ WORK IN A WORKTREE.** This is a large migration that breaks auth entirely mid-flight. Create a worktree before starting:

> cd ../<branch-name>

### Task 3: Create `convex/users.ts`

**Files:**
- Create: `convex/users.ts`

This module replaces `convex/clerkActions.ts`. It provides:
- `setCaregiverRole` — called from `acceptInvite` in `caregivers.ts`
- `setUserRole` — admin/repair utility
- `currentUser` — query for the authenticated user's record (used by server guards + client hooks)

- [ ] **Step 1: Create the file**

```ts
// convex/users.ts
import { v } from "convex/values";

import { internalMutation, query } from "./_generated/server";
import { getAuthUserId } from "@convex-dev/auth/server";

export const currentUser = query({
  args: {},
  handler: async (ctx) => {
    const userId = await getAuthUserId(ctx);
    if (!userId) return null;
    return await ctx.db.get(userId);
  },
});

export const setCaregiverRole = internalMutation({
  args: { userId: v.string() },
  handler: async (ctx, args) => {
    // userId here is the string subject from ctx.auth.getUserIdentity()
    // which equals the users table document ID
    const user = await ctx.db.get(args.userId as any);
    if (!user) throw new Error(`User not found: ${args.userId}`);
    if (user.role) {
      console.warn(
        `Skipping setCaregiverRole: user ${args.userId} already has role "${user.role}"`
      );
      return;
    }
    await ctx.db.patch(args.userId as any, { role: "caregiver" });
  },
});

export const setUserRole = internalMutation({
  args: {
    userId: v.string(),
    role: v.union(v.literal("slp"), v.literal("caregiver")),
  },
  handler: async (ctx, args) => {
    await ctx.db.patch(args.userId as any, { role: args.role });
  },
});
```

> **Note on `as any`:** The `userId` arg is a string (the `users` document ID as returned by `getAuthUserId`). Convex's type system expects `Id<"users">`. Cast with `as any` for now — once the schema is fully typed, replace with `v.id("users")` and remove the cast. This is a known pattern when the auth library owns the ID type.

// convex/auth.ts
import Google from "@auth/core/providers/google";
import Apple from "@auth/core/providers/apple";
import { Password } from "@convex-dev/auth/providers/Password";
import { convexAuth } from "@convex-dev/auth/server";

export const { auth, signIn, signOut, store } = convexAuth({
  providers: [Password, Google, Apple],
  callbacks: {
    async createOrUpdateUser(ctx, { existingUserId, profile }) {
      if (existingUserId) {
        // User already exists — don't overwrite role on re-sign-in
        return existingUserId;
      }
      // New sign-up — insert into users table
      // role defaults to null; assertSLP treats null as "slp" (new accounts are SLPs)

        email: profile.email ?? undefined,
        name: profile.name ?? undefined,
        image: profile.picture ?? undefined,
      });
    },
  },
});
```

> **Note:** The `createOrUpdateUser` callback is called on every authentication event. For existing users (`existingUserId` is non-null), return the existing ID to avoid touching the role. For new sign-ups, insert a record — role stays `undefined` (SLP by default via `assertSLP`).

const isProtectedRoute = createRouteMatcher([

  "/my-tools(.*)",
  "/settings(.*)",
  "/patients(.*)",
  "/family(.*)",
  "/sessions(.*)",
  "/billing(.*)",

// src/app/api/lib/authenticate.ts
import { convexAuthNextjsToken } from "@convex-dev/auth/nextjs/server";
import { ConvexHttpClient } from "convex/browser";

const CONVEX_URL = process.env.NEXT_PUBLIC_CONVEX_URL!;

export interface AuthResult {
  convex: ConvexHttpClient;
  userId: string | undefined;
}

export async function authenticate(): Promise<AuthResult> {
  const convex = new ConvexHttpClient(CONVEX_URL);

  try {
    const token = await convexAuthNextjsToken();
    if (token) convex.setAuth(token);
  } catch (err) {
    const msg = err instanceof Error ? err.message : "";
    console.warn("[auth] Could not retrieve Convex Auth token:", msg);
  }

  // userId is not directly available from the token without a DB roundtrip.
  // API routes that need the userId should call api.users.currentUser via convex.query().
  return { convex, userId: undefined };
}
```

// src/features/auth/components/forgot-password-card.tsx
"use client";

import { useAuthActions } from "@convex-dev/auth/react";
import { useRouter } from "next/navigation";
import { useState } from "react";

import { Button } from "@/shared/components/ui/button";
import { Input } from "@/shared/components/ui/input";

type Step = "request" | "verify";

export function ForgotPasswordCard() {
  const { signIn } = useAuthActions();
  const router = useRouter();

  const [step, setStep] = useState<Step>("request");
  const [email, setEmail] = useState("");
  const [code, setCode] = useState("");
  const [newPassword, setNewPassword] = useState("");
  const [formError, setFormError] = useState<string | null>(null);
  const [isBusy, setIsBusy] = useState(false);

  const handleRequest = async (event: React.FormEvent<HTMLFormElement>) => {
    event.preventDefault();
    setFormError(null);
    setIsBusy(true);
    try {
      await signIn("password", { email, flow: "reset" });
      setStep("verify");
    } catch (err) {
      setFormError(err instanceof Error ? err.message : "Could not send reset email.");
    } finally {
      setIsBusy(false);
    }
  };

  const handleVerify = async (event: React.FormEvent<HTMLFormElement>) => {
    event.preventDefault();
    setFormError(null);
    setIsBusy(true);
    try {
      await signIn("password", { email, code, newPassword, flow: "reset-verification" });
      router.push("/sign-in");
    } catch (err) {
      setFormError(err instanceof Error ? err.message : "Could not reset password.");
    } finally {
      setIsBusy(false);
    }
  };

  return (
    <div className="w-full max-w-[360px] rounded-xl bg-surface px-5 py-5 shadow-[0_18px_55px_rgba(26,25,23,0.1)] ring-1 ring-border/70">
      {step === "request" ? (
        <form onSubmit={handleRequest} className="space-y-3">
          <h2 className="text-lg font-semibold text-on-surface">Reset your password</h2>
          <p className="text-sm text-on-surface-variant">
            Enter your email and we&apos;ll send a reset code.
          </p>
          <Input
            type="email"
            value={email}
            onChange={(e) => setEmail(e.target.value)}
            placeholder="Email address"
            required
            className="h-12 rounded-xl border border-border bg-background"
          />
          {formError && (
            <p className="rounded-lg bg-error-container px-4 py-3 text-sm text-error">{formError}</p>
          )}
          <Button type="submit" variant="gradient" disabled={!email || isBusy} className="w-full rounded-xl text-sm font-semibold">
            Send reset code
          </Button>
        </form>
      ) : (
        <form onSubmit={handleVerify} className="space-y-3">
          <h2 className="text-lg font-semibold text-on-surface">Enter reset code</h2>
          <p className="text-sm text-on-surface-variant">
            We sent a code to {email}.
          </p>
          <Input
            type="text"
            value={code}
            onChange={(e) => setCode(e.target.value)}
            placeholder="Reset code"
            inputMode="numeric"
            autoComplete="one-time-code"
            required
            className="h-12 rounded-xl border border-border bg-background"
          />
          <Input
            type="password"
            value={newPassword}
            onChange={(e) => setNewPassword(e.target.value)}
            placeholder="New password"
            autoComplete="new-password"
            required
            className="h-12 rounded-xl border border-border bg-background"
          />
          {formError && (
            <p className="rounded-lg bg-error-container px-4 py-3 text-sm text-error">{formError}</p>
          )}
          <Button type="submit" variant="gradient" disabled={!code || !newPassword || isBusy} className="w-full rounded-xl text-sm font-semibold">
            Reset password
          </Button>
        </form>
      )}
    </div>
  );
}

## Phase 7: E2E Tests

### Task 18: Add test users to `convex/e2e_seed.ts`

**Files:**
- Modify: `convex/e2e_seed.ts`

The E2E tests need two pre-seeded users (SLP + caregiver) with known passwords. Since `useAuthActions().signIn()` is a client-side call, seeding must happen via direct Convex mutations. We insert into `users` directly (bypassing the auth flow) for test users.

> **Note:** For Convex Auth's Password provider, authenticated sessions require both a `users` row AND an `authAccounts` row linking the user to the password credential. The `authAccounts` row stores the hashed password. Inserting a `users` row alone is not enough for sign-in to work.

export const setUserRoleByEmail = internalMutation({

    email: v.string(),
    role: v.union(v.literal("slp"), v.literal("caregiver")),
  },

## Notes for Implementer

### Apple Sign-In
Apple Sign-In is wired in code but requires an Apple Developer account to work end-to-end:
1. Create a Services ID in the Apple Developer portal (`com.vocali.health.signin`)
2. Enable Sign-In with Apple capability
3. Add the Convex callback URL: `https://<deployment>.convex.site/api/auth/callback/apple`
4. Generate a private key (.p8 file)
5. Set Convex env vars: `AUTH_APPLE_ID`, `AUTH_APPLE_TEAM_ID`, `AUTH_APPLE_KEY_ID`, `AUTH_APPLE_PRIVATE_KEY`

Until those credentials exist, the Apple button will render but throw on click.

### Password Reset via Resend
The `Password` provider's `reset` flow sends email via whatever email transport is configured. If Resend is set up for `@convex-dev/auth`, it should work automatically. If not, check the `@convex-dev/auth` docs for configuring a custom email sender. The existing `convex/email.ts` + Resend integration may need to be wired to the auth library's email callback.

### Caregiver Redirect
The middleware no longer does the caregiver `/dashboard` → `/family` redirect. It now happens client-side in `dashboard-sidebar.tsx` via `useEffect`. There will be a brief flash of the dashboard before redirect. If this is unacceptable, investigate whether `@convex-dev/auth`'s middleware exposes session claims that include custom profile fields — if so, the redirect can move back to middleware.

### `authTables` Extension API
The exact syntax for extending the `users` table with a `role` field should be verified against the installed version of `@convex-dev/auth`. Run `cat node_modules/@convex-dev/auth/README.md | grep -A 20 "extend"` to find the current pattern.
