Three Approaches: Library-Optimized Bridges

How aggressively should we replace custom code with libraries?

A

Surgical Additions (Recommended)

Add 5 targeted libraries to the existing plan. Keep the current roadmap structure — just swap custom code for library equivalents where they're a clear win.

ADDS
  • @assistant-ui/react — replaces 3 custom chat components with pre-built shadcn-compatible thread UI + ExternalStoreRuntime for Convex Agent
  • shadcn resizable — replaces custom split-panel CSS in builder
  • react-error-boundary — replaces custom ErrorBoundary classes
  • usehooks-ts — replaces 4-5 custom hooks (useMediaQuery, useCountdown, useCopyToClipboard, useDebounce)
  • react-markdown + remark-gfm — chat markdown rendering (may be redundant if @assistant-ui handles this)
  • shadcn skeleton — loading states
IMPACT

~300-400 lines of custom code eliminated. Roadmap stays at 62 tasks but several shrink significantly (TASK-007, TASK-015, TASK-016, TASK-044, TASK-045). Lowest risk — well-tested libraries, drop-in replacements.

Pros

  • Minimal plan disruption
  • Each library is a clear, isolated win
  • Easy to revert if a library doesn't work
  • Chat UI is the biggest time save (~2-3 hours)

Cons

  • Doesn't question the overall scope
  • Still 62 tasks for 5 days
B

Surgical Additions + Scope Trim

Same library additions as A, but also cut scope to increase confidence of shipping in 5 days. Drop from 5 therapy tools to 3 (communication board, token board, visual schedule — the demo flow). Defer choice board and first-then board to a stretch goal.

CHANGES
  • All library additions from Approach A
  • Drop TASK-023 (choice board) and TASK-024 (first-then board) to stretch goals
  • Reduce templates from 8 to 6 (drop 2 categories)
  • Simplify landing page — hero + CTA only, skip How It Works animation
  • ~52 tasks instead of 62

Pros

  • More realistic for 5-day timeline
  • 3 polished tools > 5 rough tools for demo
  • Freed time goes to polish and reliability

Cons

  • Judges may notice fewer tool types
  • Reduces "wow" factor slightly
C

Maximum Velocity — Template Fork

Start from Vercel's open-source AI chatbot template (Next.js + AI SDK + shadcn), rip out its auth/database layer, wire in Convex Agent, and bolt on therapy tools. Pre-built chat UI, layout, streaming, markdown — all included.

CHANGES
  • Fork vercel/ai-chatbot as the starting point
  • Replace its database (Postgres) with Convex
  • Replace its auth (NextAuth) with nothing (add Clerk later)
  • Add therapy tool components + tool renderer on top
  • Potentially save Phase 0 setup time entirely

Pros

  • Fastest possible start — chat UI works day 1
  • Battle-tested layout, streaming, markdown

Cons

  • Ripping out Postgres/NextAuth may cost more time than starting fresh
  • Template's opinions may fight your architecture (VSA, Convex Agent)
  • Forked code is harder to maintain long-term
  • High risk of hidden incompatibilities
