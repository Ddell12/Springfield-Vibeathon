# Clerk → Convex Auth Migration Design

**Date:** 2026-04-03  
**Status:** Approved  
**Motivation:** Simplification (remove external service dependency) + Control (fully owned auth flow, custom UI, no Clerk branding)

---

## Goals

- Remove all `@clerk/*` dependencies from the codebase
- Auth runs entirely inside the Convex deployment — no external auth service
- Preserve the existing sign-in UI components (adapt hooks only)
- Support: email + password, Google OAuth, Apple Sign-In
- Clean break — no user data migration from Clerk (user base is small/dev)
- Rebuild E2E test fixtures to use real form-fill sign-in against seeded Convex users

---

## Out of Scope

- Migrating existing Clerk users (clean break accepted)
- Magic link / OTP sign-in (not requested)
- Any visual redesign of the sign-in / sign-up screens

---

## Architecture Overview

### Replacement Map

| Current (Clerk) | Replaced With |
|---|---|
| `<ClerkProvider>` in `layout.tsx` | `<ConvexAuthProvider>` from `@convex-dev/auth/react` |
| `ConvexProviderWithClerk` in `providers.tsx` | Removed — `ConvexAuthProvider` handles both |
| `clerkMiddleware` in `proxy.ts` | `convexAuthNextjsMiddleware` + `isAuthenticatedNextjs` |
| `currentUser()` in server-role-guards | Convex Auth Next.js server helpers + `ctx.db` role read |
| `auth()` + `getToken` in `authenticate.ts` | Convex Auth session token helpers |
| `convex/auth.config.ts` (Clerk JWT domain) | Auto-generated by `npx @convex-dev/auth` |
| `convex/clerkActions.ts` (Clerk REST API) | `convex/users.ts` — `internalMutation` writing `role` to `authUsers` |
| Role in `public_metadata` JWT claim | `role` field on extended `authUsers` table |
| `@clerk/testing/playwright` in E2E | Standard Playwright form-fill against seeded Convex users |

### Key Architectural Shift

Currently roles flow: **Clerk cloud (publicMetadata) → Clerk JWT claim → Convex identity → `getAuthRole()`**.

After migration: **`authUsers.role` field in Convex → direct `ctx.db` read in `getAuthRole()`**.

No external API call, no JWT template configuration, no `CLERK_SECRET_KEY` in Convex env.

## Section 2: Schema Changes

### `convex/schema.ts`

Add `authTables` with a `role` profile extension. Spread before all existing tables.

```ts
import { authTables } from "@convex-dev/auth/server";
import { v } from "convex/values";

export default defineSchema({
  ...authTables({
    profile: {
      role: v.optional(v.union(v.literal("slp"), v.literal("caregiver"))),
    },
  }),
  // ...all existing tables unchanged
});
```

This adds four tables: `authUsers`, `authAccounts`, `authSessions`, `authVerificationCodes`.  
The `role` field on `authUsers` replaces Clerk's `publicMetadata.role`.

### `convex/auth.ts` — provider configuration

```ts
import { convexAuth } from "@convex-dev/auth/server";
import { Password } from "@convex-dev/auth/providers/Password";
import Google from "@auth/core/providers/google";
import Apple from "@auth/core/providers/apple";

export const { auth, signIn, signOut, store } = convexAuth({
  providers: [Password, Google, Apple],
});
```

## Section 3: Backend Auth Layer

### `convex/auth.config.ts`

Auto-generated by `npx @convex-dev/auth`. Do not hand-edit after init.

### `convex/lib/auth.ts` — role pipeline rewrite

`getAuthRole` currently parses `identity.public_metadata` (a Clerk JWT claim with dual string/object format). After migration it reads directly from `authUsers`:

```ts
export async function getAuthRole(
  ctx: QueryCtx | MutationCtx,
): Promise<UserRole | null> {
  const userId = await getAuthUserId(ctx);
  if (!userId) return null;
  const user = await ctx.db.get(userId as Id<"authUsers">);
  return (user?.role as UserRole) ?? null;
}
```

All other functions in `convex/lib/auth.ts` (`assertSLP`, `assertCaregiverAccess`, `assertPatientAccess`, `assertSessionOwner`) are unchanged — they already call `getAuthRole` and `getAuthUserId`.

### `convex/clerkActions.ts` → `convex/users.ts`

Delete `convex/clerkActions.ts`. Create `convex/users.ts` with an `internalMutation` for role assignment:

```ts
export const setCaregiverRole = internalMutation({
  args: { userId: v.id("authUsers") },
  handler: async (ctx, args) => {
    const user = await ctx.db.get(args.userId);
    if (!user) throw new Error("User not found");
    if (user.role) {
      console.warn(`Skipping setCaregiverRole: user already has role "${user.role}"`);
      return;
    }
    await ctx.db.patch(args.userId, { role: "caregiver" });
  },
});

export const setUserRole = internalMutation({
  args: { userId: v.id("authUsers"), role: v.union(v.literal("slp"), v.literal("caregiver")) },
  handler: async (ctx, args) => {
    await ctx.db.patch(args.userId, { role: args.role });
  },
});
```

All callers of `internal.clerkActions.setCaregiverRole` in `convex/caregivers.ts` are updated to `internal.users.setCaregiverRole`.

## Section 4: Frontend Wiring

### `src/app/layout.tsx`

Remove `<ClerkProvider>`. `ConvexAuthProvider` replaces both Clerk and Convex providers:

```tsx
import { ConvexAuthProvider } from "@convex-dev/auth/react";
import { ConvexReactClient } from "convex/react";

const convex = new ConvexReactClient(process.env.NEXT_PUBLIC_CONVEX_URL!);

// In RootLayout body:
<ConvexAuthProvider client={convex}>
  <ThemeProvider attribute="class" defaultTheme="light" enableSystem>
    {children}
    <Toaster />
  </ThemeProvider>
</ConvexAuthProvider>
```

### `src/core/providers.tsx`

Deleted entirely. Its only job was `ConvexProviderWithClerk`. `ConvexAuthProvider` replaces it.

### `src/proxy.ts` (Next.js middleware)

Replace `clerkMiddleware` with `convexAuthNextjsMiddleware`:

```ts
import {
  convexAuthNextjsMiddleware,
  createRouteMatcher,
  isAuthenticatedNextjs,
  nextjsMiddlewareRedirect,
} from "@convex-dev/auth/nextjs/server";

const isProtectedRoute = createRouteMatcher([
  "/dashboard(.*)", "/my-tools(.*)", "/settings(.*)",
  "/patients(.*)", "/family(.*)", "/sessions(.*)",
  "/billing(.*)", "/speech-coach(.*)",
]);

const isPublicApiRoute = createRouteMatcher([
  "/api/tool/(.*)", "/family/(.*)/play/manifest.json", "/apps/(.*)",
]);

export default convexAuthNextjsMiddleware(async (request, { convexAuth }) => {
  if (isPublicApiRoute(request)) return;

  // Caregiver → /family redirect
  // Role is embedded in Convex Auth session token, readable via convexAuth
  // Implementation: fetch role from session claims or short-circuit to client-side redirect

  if (isProtectedRoute(request) && !(await isAuthenticatedNextjs(convexAuth))) {
    return nextjsMiddlewareRedirect(request, "/sign-in");
  }
});

export const config = {
  matcher: [
    "/((?!_next|api/tool/|apps/|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)",
    "/(api(?!/tool/))(.*)",
    "/(trpc)(.*)",
  ],
};
```

> **Note:** The caregiver role-based redirect in middleware requires reading the role from the Convex Auth session. The implementation detail (whether to embed role in the session JWT or do a short Convex query) should be resolved during implementation — the simplest correct approach is to do the caregiver redirect client-side in the dashboard layout (as a fallback) if the session token doesn't embed role claims.

### `src/app/api/lib/authenticate.ts`

Replace Clerk's `auth()` + `getToken` with Convex Auth's `convexAuthNextjsToken`:

```ts
import { convexAuthNextjsToken } from "@convex-dev/auth/nextjs/server";
import { ConvexHttpClient } from "convex/browser";

export async function authenticate(): Promise<AuthResult> {
  const convex = new ConvexHttpClient(process.env.NEXT_PUBLIC_CONVEX_URL!);
  const token = await convexAuthNextjsToken();
  if (token) convex.setAuth(token);
  return { convex, userId: undefined }; // userId resolved via Convex query if needed
}
```

### `src/features/auth/lib/server-role-guards.ts`

Replace `currentUser()` with Convex Auth's server helpers + a Convex HTTP query for the role:

```ts
import { convexAuthNextjsToken } from "@convex-dev/auth/nextjs/server";
import { fetchQuery } from "convex/nextjs";
import { api } from "@/convex/_generated/api";
import { redirect } from "next/navigation";

export async function requireSlpUser() {
  const token = await convexAuthNextjsToken();
  if (!token) redirect("/sign-in");

  const user = await fetchQuery(api.users.currentUser, {}, { token });
  if (!user) redirect("/sign-in");
  if (user.role === "caregiver") redirect("/family");

  return user;
}
```

This requires a new `api.users.currentUser` query in `convex/users.ts`.

## Section 5: Sign-in UI Adaptation

### `src/features/auth/components/claude-sign-in-card.tsx`

Replace Clerk hooks with `useAuthActions` from `@convex-dev/auth/react`:

```tsx
import { useAuthActions } from "@convex-dev/auth/react";

const { signIn } = useAuthActions();

// Email + password:
await signIn("password", { email, password, flow: "signIn" }); // or "signUp"

// Google OAuth:
await signIn("google");

// Apple Sign-In:
await signIn("apple");
```

Visual layout, `auth-art-panel`, error states, and loading states are unchanged.

### Sign-in / sign-up pages

Remove Clerk's `<SignIn />` / `<SignUp />` pre-built components. Pages become thin wrappers around `claude-sign-in-card.tsx` with a `flow` prop.

### UserButton / sign-out

Replace Clerk's `<UserButton>` with a custom component using:
- `useCurrentUser()` — a hook wrapping `useQuery(api.users.currentUser)` for avatar/name
- `useAuthActions().signOut()` — for the sign-out action

### Password reset

Implemented via `@convex-dev/auth`'s `Password` provider reset flow + the existing Resend integration:
1. User submits email → `signIn("password", { email, flow: "reset" })` sends a code via Resend
2. User submits code + new password → `signIn("password", { email, code, newPassword, flow: "reset-verification" })`

## Section 7: E2E Test Fixtures

### Test user seeding

Extend `convex/e2e_seed.ts` (a Convex action callable via `npx convex run e2e_seed`) to create test users using Convex Auth's internal mutation APIs. Because `signIn` is a client-side action, seeding must go through internal mutations — specifically `@convex-dev/auth`'s internal `createUser` or by calling `ctx.runMutation(internal.auth.store, ...)` with the appropriate provider args.

The exact seeding API depends on the `@convex-dev/auth` version; during implementation, check the library's test/seed examples. The outcome must be:

1. Two `authUsers` documents: one with `role: "slp"`, one with `role: "caregiver"`
2. Corresponding `authAccounts` documents with hashed passwords (using `Password` provider)
3. Test credentials stored in `.env.local` / CI env vars under the existing names (`E2E_SLP_EMAIL`, `E2E_SLP_PASSWORD`, `E2E_CAREGIVER_EMAIL`, `E2E_CAREGIVER_PASSWORD`)

### `tests/e2e/fixtures.ts`

Replace `@clerk/testing/playwright` with a `signInAs` helper:

```ts
async function signInAs(page: Page, email: string, password: string) {
  await page.goto("/sign-in");
  await page.fill('[name="email"]', email);
  await page.fill('[name="password"]', password);
  await page.click('[type="submit"]');
  await page.waitForURL(/\/(dashboard|family)/);
}

export const slpPage = base.extend<{ page: Page }>({
  page: async ({ page }, use) => {
    await signInAs(page, process.env.E2E_SLP_EMAIL!, process.env.E2E_SLP_PASSWORD!);
    await use(page);
  },
});

export const caregiverPage = base.extend<{ page: Page }>({
  page: async ({ page }, use) => {
    await signInAs(page, process.env.E2E_CAREGIVER_EMAIL!, process.env.E2E_CAREGIVER_PASSWORD!);
    await use(page);
  },
});

export const authedPage = slpPage; // backward-compatible alias
```

Env vars (`E2E_SLP_EMAIL`, `E2E_SLP_PASSWORD`, `E2E_CAREGIVER_EMAIL`, `E2E_CAREGIVER_PASSWORD`) keep the same names.

### `tests/e2e/auth.spec.ts`

Rewrite to cover:
- Email + password sign-up → redirects to `/dashboard`
- Sign-out → redirects to `/sign-in`
- Protected route access when unauthenticated → redirects to `/sign-in`
- Caregiver sign-in → redirects from `/dashboard` to `/family`

## Open Questions

- **Caregiver role-based redirect in middleware:** Convex Auth's Next.js middleware does not automatically embed custom profile fields (like `role`) in the session token. The caregiver `/dashboard` → `/family` redirect may need to move to a client-side check in the dashboard layout rather than middleware. Confirm during implementation.
- **Apple Sign-In:** Requires Apple Developer account setup before the provider works end-to-end. Can be wired in code first and enabled once credentials are ready.
