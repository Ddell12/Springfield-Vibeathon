# Workflow Builder

Generates, typechecks, and creates PRs for new Trigger.dev tasks from natural language descriptions. This is Aura's self-programming capability for workflow automation.

## Workflow

When asked to create a new automation, follow these steps exactly:

### Step 1: Parse the Request

Extract from the natural language description:

- **Task ID**: kebab-case identifier (e.g., `hn-checker`, `email-digest`)
- **Export name**: camelCase version (e.g., `hnChecker`, `emailDigest`)
- **Schedule**: Cron expression if recurring (e.g., `0 * * * *` for hourly)
- **Needs route?**: Does it need a custom daemon endpoint?
- **Needs Convex storage?**: Does it need to store results in a Convex table?
- **Template**: Which template to use (A: on-demand, B: scheduled, C: daemon-calling)

### Step 2: Create a Feature Branch

```bash
git checkout -b workflow/<task-id> main
```

### Step 3: Generate the Task File

Create `src/trigger/<task-id>.ts` using the appropriate template from `references/task-templates.md`:

- **Template A (On-demand)**: For tasks triggered manually or by other tasks/webhooks
- **Template B (Scheduled)**: For recurring cron-based tasks that run through the agent
- **Template C (Daemon-calling)**: For scheduled tasks that call a specific daemon route

Fill in the template variables:

- `{{taskId}}` — the kebab-case task ID
- `{{taskExport}}` — the camelCase export name
- `{{cronPattern}}` — cron expression (Template B/C only)
- `{{promptTemplate}}` — the agent prompt or skill to invoke

### Step 4: Generate Supporting Files (if needed)

**If route needed** (Template C or custom webhook processing):

- Create handler in `src/scheduling/routes/<task-id>.ts`
- Register in `src/scheduling/trigger-server.ts`

**If Convex storage needed**:

- Add table definition to `convex/schema.ts`
- Create CRUD functions in `convex/workflow/<task-id>.ts`

### Step 5: Update Barrel Export

Add to `src/trigger/index.ts`:

```typescript
export { {{taskExport}} } from "./{{task-id}}.js";
```

### Step 6: Typecheck

```bash
npm run typecheck
```

If typecheck fails:

1. Read the error output
2. Fix the issue
3. Re-run typecheck
4. Maximum 3 attempts — if still failing, stop and report errors

### Step 7: Commit and PR

```bash
git add src/trigger/<task-id>.ts src/trigger/index.ts
# Add any other generated files
git commit -m "feat(trigger): add <task-id> workflow task

<brief description of what the task does>"
git push -u origin workflow/<task-id>
gh pr create --title "Add <task-id> workflow" --body "## Summary
- <description of the automation>
- Template: <A/B/C>
- Schedule: <cron or on-demand>

Generated by workflow-builder skill."
```

### Step 8: Create Schedule (if applicable)

For scheduled tasks (Template B/C), optionally create the Trigger.dev schedule:

```bash
# The schedule is defined in the task file itself via schedules.task()
# It will be created automatically on deploy
```

For on-demand tasks that need a Convex-managed schedule:

```bash
npx convex run scheduling/triggerBridge:createSchedule \
  '{"cron": "<pattern>", "externalId": "<task-id>", "prompt": "", "timezone": "America/Chicago"}'
```

## Examples

**"Create a task that checks Hacker News every hour"**

- Task ID: `hn-checker`
- Template B (scheduled)
- Cron: `0 * * * *`
- Prompt: "Check the top 5 Hacker News stories and summarize any relevant to AI/ML"

**"Build an automation that processes Stripe webhooks"**

- Task ID: `stripe-processor`
- Template A (on-demand, triggered by webhook)
- Register webhook source via webhook-manager skill

**"Make a workflow that backs up my vault daily"**

- Task ID: `vault-backup`
- Template C (daemon-calling)
- Cron: `0 2 * * *` (2 AM)
- Route: `/vault-backup`

## Important Conventions

- **ESM imports**: All local imports must use `.js` extensions
- **File location**: Task files go in `src/trigger/`
- **Naming**: File name matches task ID (kebab-case), export is camelCase
- **No cross-slice imports**: Tasks in `src/trigger/` can import from `./` (trigger slice) and `../../convex/_generated/api.js`
- **Branch flow**: Always branch + PR, never commit directly to main
- **Auto-deploy**: Merging to main triggers auto-deploy via GitHub webhook
